Architettura AWS Cloud - Kitchen Manager

Report: Migrazione e Implementazione Cloud-Native

Il presente documento illustra l'architettura cloud ideata per il progetto Kitchen Manager (Stack: React, Vite, Django, PostgreSQL) su infrastruttura Amazon Web Services. L'obiettivo dell'implementazione è mappare un ambiente monolitico locale verso un set di componenti cloud-native per garantire scalabilità, sicurezza ed alta disponibilità.

1. Livello Frontend (React + Vite)

  • Il bundle statico autogenerato dal framework frontend risiede all'interno di Amazon S3, configurato specificatamente in modalità Static Website Hosting.
  • La distribuzione globale dei contenuti e l'accelerazione delle tempistiche di risposta sono garantite attraverso Amazon CloudFront (CDN), che offre caching a livello globale, crittografia avanzata nativa (HTTPS via ACM) e un primo layer difensivo mediante AWS WAF (Web Application Firewall).

2. Livello Backend (Compute & PaaS)

  • L'applicativo backend Python/Django richiede risorse computazionali in grado di scalare dinamicamente nei picchi di carico. Per l'hosting è impiegato AWS Elastic Beanstalk, una nota componente Platform as a Service (PaaS).
  • Il servizio orchestra la fornitura delle istanze Amazon EC2, automatizzando il bilanciamento del carico in ingresso per reagire in tempo reale alle variazioni di traffico, assicurando scalabilità orizzontale e massima resilienza.

3. Livello Dati (Relational Database)

  • Tutte le transazioni e i dati di gestione sono delegati al servizio gestito Amazon RDS (PostgreSQL Engine).
  • RDS si fa carico dell'automazione integrale dei backup e della manutenzione. La comunicazione Backend-Database avviene sul perimetro di una singola Virtual Private Cloud (VPC), con una rigorosa segregazione via Security Groups organizzati in chiave Least Privilege, esponendo la porta 5432 esplicitamente al solo gruppo di sicurezza relativo ai layer computazionali di backend.

4. Integrazione AI e Gestione dei Segreti (Decoupling)

  • Al fine di evitare restrizioni o blocchi sui thread dei server Web principali, le funzionalità di elaborazione del linguaggio supportate dall'AI (come l'AI Google Gemini) vengono eseguite fuori dal flusso di richiesta dell'interfaccia utente. Le chiamate sono processate da una coda Amazon SQS che scatena le esecuzioni mediante AWS Lambda.
  • Rispettando le norme Enterprise zero-trust, le chiavi identificative associate ai modelli AI non vengono hardcodate internamente alle funzioni analizzatrici, ma prelevate contestualmente alle esecuzioni dalle istanze sicure di AWS Secrets Manager.

5. Compliance e Security Posture

  • L'intera architettura in esenzione si rifà alle stringenti normative del Well-Architected Framework delineato da AWS, e segue le corrette attribuzioni descritte nel Shared Responsibility Model.
  • Le configurazioni identitarie seguono policy modulari in ecosistema AWS IAM, scongiurando l'abuso dei privilegi da parte di singole classi dell'applicativo (es. operatori in front e backofice).
